# Plan 1-4: Typer CLI Application

## Objective
Build the main entry point for the user via a Typer-based CLI. The CLI will provide commands to start the daemon in the background (`agent start --daemon`), stop it, view its status, and tail its logs, complete with Rich terminal formatting.

## Code Patterns (Model Resilience)
Reference these existing files for code style:
- `agent/logger.py` — Use the logger initialization pattern.

Conventions:
- File naming: `cli.py` for commands, `main.py` for entry point.
- Error handling: Use `typer.Exit(code=...)` on fatal command-line errors.
- Types: Typer strictly requires type hints for CLI arguments (`typer.Option`, `typer.Argument`).

## Tasks

<task type="auto">
  <name>Build Typer Commands</name>
  <files>agent/cli.py, main.py, tests/test_cli.py</files>
  <action>
    1. Import `typer`, and `Console` from `rich.console`. Install rich formatting via `rich.print`.
    2. Create a Typer app: `app = typer.Typer()`.
    3. Define `@app.command() def start(daemon: bool = typer.Option(False, "--daemon", "-d", help="Run in background"))`.
    4. In `start`, if `daemon` is true, invoke the `DaemonWatcher` logic in detached mode (e.g., via multiprocessing or standard subprocess `Popen`), otherwise run attached.
    5. Define `@app.command() def stop()`: Identify the running daemon process (perhaps via a pidfile saved in `~/.sg_agent/agent.pid`) and send `SIGTERM`.
    6. Define `@app.command() def status()`: Check the pidfile, ping the process, and print a Rich-styled status message (green for running, red for stopped).
    7. Define `@app.command() def logs()`: Open and tail the `agent.log` file formatting output via Rich.
    8. Write unit tests to execute Typer commands using `typer.testing.CliRunner`.
  </action>
  <verify>pytest tests/test_cli.py</verify>
  <done>All commands (start, stop, status, logs) are executable, handle parameters correctly, and generate colorful output.</done>
</task>

## Must-Haves
- `start --daemon` launches the background process.
- `status` correctly identifies running state.
- `pidfile` management must safely handle stale files if the daemon crashed hard.
